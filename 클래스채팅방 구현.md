# 클래스채팅방 구현

## 가능 여부

클래스별 실시간 채팅방 구현은 가능합니다.

현재 1:1 메시지 기능이 리얼타임으로 구현되어 있다면, 같은 구조를 클래스 단체 채팅으로 확장할 수 있습니다.

## 개념 비교

### 1:1 채팅

- 대화방 기준: 나 + 상대 1명
- 메시지 저장 기준: `conversation_id`
- 참여자 수: 2명

### 클래스 채팅

- 대화방 기준: 클래스 1개
- 메시지 저장 기준: `class_id`
- 참여자 수: 클래스 개설자 + 승인된 신청자 여러 명

## 기본 구조

클래스 채팅방은 클래스 하나당 하나씩 연결됩니다.

예시:

```txt
클래스 A
- 개설자
- 승인된 신청자 1
- 승인된 신청자 2
- 승인된 신청자 3

=> 클래스 A 채팅방 입장 가능
```

## 필요한 기능

1. 클래스별 채팅 메시지 저장
2. 클래스 개설자와 승인된 신청자만 입장 가능
3. Supabase Realtime으로 새 메시지 즉시 반영
4. 클래스 상세 페이지에서 채팅방으로 이동
5. 채팅방에서 메시지 작성/조회

## 필요한 DB 예시

```sql
CREATE TABLE class_chat_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  class_id UUID NOT NULL REFERENCES classes(id) ON DELETE CASCADE,
  sender_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

권한 정책은 다음 기준이 필요합니다.

- 클래스 개설자는 조회/작성 가능
- 승인된 신청자는 조회/작성 가능
- 승인되지 않은 사용자는 접근 불가

## 구현 난이도

기본 텍스트 실시간 채팅만 구현하면 난이도는 `중간`입니다.

이미 1:1 리얼타임 채팅 구조가 있으므로 다음 부분을 재사용할 수 있습니다.

- 메시지 목록 UI
- 메시지 전송 API
- Supabase Realtime 구독 방식
- 새 메시지 화면 반영 로직

## 어려운 부분

1. 권한 체크

승인된 신청자만 채팅방에 들어오게 해야 합니다.

2. 신청 취소/승인 취소 처리

신청자가 취소되거나 승인 취소되면 채팅방 접근을 막아야 합니다.

3. 알림 처리

새 메시지 알림까지 붙이면 복잡도가 올라갑니다.

4. 읽음 처리

누가 읽었는지까지 표시하려면 별도 테이블이 필요합니다.

5. 이미지/파일 전송

텍스트보다 구현 범위가 커집니다.

## 추천 구현 순서

1. `class_chat_messages` 테이블 추가
2. 클래스 상세 페이지에 `단체채팅` 버튼 추가
3. 클래스 채팅방 페이지 생성
4. 승인된 신청자/개설자만 접근 가능하게 API 권한 체크
5. 기존 1:1 채팅 UI를 참고해서 텍스트 메시지 전송 구현
6. Supabase Realtime 구독 연결
7. 이후 알림/읽음/이미지 전송 확장

## 결론

클래스 채팅방 구현은 가능합니다.

기본 텍스트 실시간 단체 채팅은 `중간 난이도`입니다.

읽음 처리, 알림, 이미지 전송, 참여자 관리까지 포함하면 `중상 난이도`로 봐야 합니다.
